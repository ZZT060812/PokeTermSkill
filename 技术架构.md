# Remote AI Terminal — 技术架构文档

## 概述

Remote AI Terminal 是一个轻量级的远程终端 + 文件管理系统。核心场景：主力机在家跑 AI 任务（Claude Code / 模型训练），用户出门在外用手机/平板/轻薄本的浏览器就能实时查看终端输出、操作文件，无需安装任何客户端。

## 设计目标

1. **零客户端安装** — 任何有浏览器的设备都能用
2. **真实终端体验** — PTY 伪终端，支持交互式程序（vim、htop、Claude Code 等）
3. **单连接复用** — 终端 IO 和文件操作共用一个 WebSocket
4. **移动端可用** — 响应式布局，手机和平板操作舒适
5. **安全边界清晰** — token 认证 + 路径沙箱
6. **断联重连** — 网络切换自动恢复，回放丢失输出，用户无感

## 通信协议

### 消息格式

所有消息均为 JSON，通过 WebSocket 文本帧传输。每条消息至少包含 `type` 字段：

```json
{"type": "消息类型", ...}
```

### 生命周期

```
客户端                        服务端
  │                             │
  │── WebSocket 握手 ──────────►│
  │                             │
  │── {"type":"auth","token":"x"}──►│  认证阶段
  │◄── {"type":"auth","ok":true} ──│
  │                             │
  │── {"type":"term.init",...} ──►│  初始化终端
  │                             │── spawn PTY shell
  │◄── {"type":"term.output",...}─│  开始推送输出
  │                             │
  │── {"type":"term.input",...} ──►│  用户输入
  │── {"type":"fs.list",...} ────►│  文件操作
  │◄── {"type":"fs.list",...} ────│
  │                             │
  │── WS Close ────────────────►│  断开连接
  │          (PTY 继续运行)       │
```

### 终端消息

#### term.init — 初始化 / 调整终端大小

```json
{
  "type": "term.init",
  "cols": 120,
  "rows": 40
}
```
客户端建立连接后发送，服务端据此设置 PTY 窗口大小。

#### term.output — 终端输出（服务端 → 客户端）

```json
{
  "type": "term.output",
  "data": "SGVsbG8gV29ybGQ="
}
```
`data` 为 Base64 编码的终端原始输出。原始字节（含 ANSI 转义序列、颜色码、控制字符）不便通过 JSON 直接传输，统一 Base64 编码。客户端解码后直接写入 xterm.js。

#### term.input — 终端输入（客户端 → 服务端）

```json
{
  "type": "term.input",
  "data": "bHMgLWxhCg=="
}
```
同样是 Base64 编码。客户端监听 xterm.js 的 `onData` 事件，将用户按键 Base64 编码后发送。

#### term.resize — 调整终端大小

```json
{
  "type": "term.resize",
  "cols": 100,
  "rows": 30
}
```
当浏览器窗口大小变化或手机旋转屏幕时触发，服务端调用 `pty.setWinSize()` 同步 PTY 窗口大小。

### 文件操作消息

#### fs.list — 列出目录

```json
// 请求
{"type": "fs.list", "path": "/Users/zengzitong/project/src"}

// 响应
{
  "type": "fs.list",
  "path": "/Users/zengzitong/project/src",
  "files": [
    {"name": "main.py", "isDir": false, "size": 1234, "mtime": 1718000000},
    {"name": "utils",   "isDir": true,  "size": 0,    "mtime": 1717900000}
  ],
  "error": null
}
```

#### fs.read — 读取文件内容

```json
// 请求
{"type": "fs.read", "path": "/Users/zengzitong/project/src/main.py"}

// 响应
{
  "type": "fs.read",
  "path": "/Users/zengzitong/project/src/main.py",
  "content": "import os\n\ndef main():\n    ...",
  "truncated": false,
  "error": null
}
```
大于 1MB 的文件截断返回，`truncated` 标记为 `true`。

#### fs.write — 写入文件

```json
// 请求
{"type": "fs.write", "path": "/Users/zengzitong/project/test.txt", "content": "Hello"}

// 响应
{"type": "fs.result", "ok": true, "error": null}
```

#### fs.delete — 删除文件/目录

```json
// 请求
{"type": "fs.delete", "path": "/Users/zengzitong/project/temp"}

// 响应
{"type": "fs.result", "ok": true, "error": null}
```

#### fs.mkdir — 创建目录

```json
// 请求
{"type": "fs.mkdir", "path": "/Users/zengzitong/project/newdir"}

// 响应
{"type": "fs.result", "ok": true, "error": null}
```

#### term.replay — 重连回放（服务端 → 客户端）

```json
{
  "type": "term.replay",
  "data": "SGVsbG8gV29ybGQK..."
}
```
仅在客户端重连后发送。服务端将环形缓冲区中的历史输出（Base64 编码）批量推送给客户端，xterm.js 写入后客户端终端恢复到断线前的视觉状态。回放完成后无缝切换到实时 `term.output`。

### 认证消息

```json
// 请求
{"type": "auth", "token": "a1b2c3d4e5f6"}

// 成功
{"type": "auth", "ok": true}

// 失败
{"type": "auth", "ok": false, "error": "Invalid token"}
```
WebSocket 建立后首条消息必须为认证消息，未认证前拒绝所有其他消息。

## 断联重连机制

### 问题场景

- 手机切换基站 / WiFi ↔ 蜂窝
- 屏幕熄屏后浏览器暂停 WebSocket
- 隧道、电梯等短暂的信号丢失
- 主力机短暂休眠后唤醒
- 用户将浏览器切到后台长时间后再切回

### 客户端重连策略

```
断开检测 (onclose / onerror)
       │
       ▼
  等待 delay 秒 ──► 新建 WebSocket
       │               │
       │          ┌────┴────┐
       │          │ 成功     │ 失败
       │          ▼         │
       │    发送 auth       │
       │    发送 term.init  │
       │    接收 term.replay│
       │    回到实时模式     │
       │                    │
       └── delay × 2 ──────┘  (指数退避)
```

退避策略：

| 重试次数 | 延迟 | 说明 |
|---------|------|------|
| 1 | 1s ± 200ms 抖动 | 快速恢复 |
| 2 | 2s ± 400ms | |
| 3 | 4s ± 800ms | |
| 4 | 8s ± 1.6s | |
| 5+ | 15s ± 3s | 封顶，不再增长 |

随机抖动避免多个客户端同时打过来（thundering herd）。

### 服务端输出缓冲区

`PtyManager` 维护一个环形缓冲区（ring buffer）：

```
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ A │ B │ C │ D │ E │   │   │   │  ← 已填充
└───┴───┴───┴───┴───┴───┴───┴───┘
  ↑               ↑
  tail            head
  
新输出写入 head，head 追尾时 tail 前移（淘汰老数据）
```

参数：
- **容量上限**：500 条输出块（每条最多 8KB），总计约 4MB 上限
- **淘汰策略**：FIFO，先写入的先淘汰
- **内容**：原始字节（写入 PTY stdout 的原始数据，含 ANSI 转义序列）

### 回放流程

```
客户端重连成功
       │
       ▼
  发送 auth (token 本地缓存，自动发送)
       │
       ▼
  发送 term.init
       │
       ▼
服务端: 取出环形缓冲区全部内容
       │
       ▼
  发送 term.replay { data: base64(全部缓冲区拼接) }
       │
       ▼
客户端: Base64 解码 → xterm.write(data)
       │  终端视觉状态恢复到断线前
       ▼
服务端: 后续输出通过 term.output 正常实时推送
```

### 边界情况

| 场景 | 处理方式 |
|------|---------|
| 断线期间缓冲区被填满多次 | 只回放最新 500 条，中间有缺口，终端会有跳跃感但不会崩溃 |
| 断线超过 PTY 进程生命周期 | PTY 退出时广播 `term.exit`，客户端重连后提示"会话已结束" |
| 新客户端首次连接（非重连） | 不发送 `term.replay`，避免刷屏 |
| 多个客户端同时重连 | 各自独立回放，互不影响 |

### 首次连接 vs 重连的区分

服务端用 `term.init` 消息中的 `fresh` 字段区分：

```json
// 首次连接
{"type": "term.init", "cols": 120, "rows": 40, "fresh": true}
// → 服务端不发 term.replay

// 重连
{"type": "term.init", "cols": 120, "rows": 40, "fresh": false}
// → 服务端发送 term.replay
```

### TerminalSocketHandler.java — 消息路由

继承 `TextWebSocketHandler`，核心逻辑：

```
handleTextMessage()
  │
  ├── 未认证? ──► 只处理 auth 消息，其余丢弃
  │
  ├── type == "term.init" ──► fresh==true? 跳过回放 :
  │                           ptyManager.getReplayData() → send term.replay
  │                           然后 ptyManager.resize()
  ├── type == "term.input" ──► ptyManager.write()
  ├── type == "term.resize"──► ptyManager.resize()
  │
  ├── type == "fs.list"   ──► fileHandler.list()  → sendMessage()
  ├── type == "fs.read"   ──► fileHandler.read()  → sendMessage()
  ├── type == "fs.write"  ──► fileHandler.write() → sendMessage()
  ├── type == "fs.delete" ──► fileHandler.delete() → sendMessage()
  └── type == "fs.mkdir"  ──► fileHandler.mkdir() → sendMessage()
```

维护一个 `Set<WebSocketSession>` 保存所有已认证连接，广播终端输出。

### 为什么是 PTY 而不是 ProcessBuilder

这是整个方案最关键的架构决策。

**ProcessBuilder pipe 方式：**
```java
Process p = new ProcessBuilder("bash").start();
p.getInputStream();   // stdout
p.getOutputStream();  // stdin
```
问题：`sudo`、`ssh`、`passwd` 等命令要求 `/dev/tty` 为真实终端设备。pipe 模式下 `isatty()` 返回 false，程序拒绝交互或报错 `no tty present`。

**pty4j PTY 方式：**
```java
PtyProcess pty = PtyProcess.exec("bash", ...);
// 操作系统分配真实的 /dev/ttys00X 设备
// isatty() → true
// stty 属性完全可用（echo 开关、信号发送等）
```

PTY 在 OS 层面创建了 `/dev/ttys00X` 设备节点，shell 和所有子进程（包括 `sudo`、`vim`、`ssh`、`Claude Code` 等）认为自己运行在真实终端上，行为与主力机键盘显示器完全一致。

**双向数据流：**
```
                        服务端
客户端 (浏览器)           ┌──────────────────────────┐
┌──────────────┐         │                          │
│ xterm.js     │         │  PtyManager              │
│              │         │  ┌────────────────────┐  │
│ onData ──────┼─Base64──┼─► write(stdin)        │  │
│   (键盘输入)  │  WS    │  │        │            │  │
│              │        │  │   ┌────▼─────┐      │  │
│              │        │  │   │  PTY      │      │  │
│              │        │  │   │ /dev/tty  │      │  │
│ write() ◄────┼─Base64─┼──│ stdout│     │      │  │
│   (终端输出)  │  WS    │  │   └────┬─────┘      │  │
│              │        │  │        │            │  │
└──────────────┘        │  │  read thread ───────┼──┤
                        │  │  读取线程循环读stdout │  │
                        │  └────────────────────┘  │
                        │                          │
                        │  完全双向、实时、全双工    │
                        └──────────────────────────┘
```

交互式程序完整支持清单：

| 程序 | 需求 | PTY 支持 |
|------|------|---------|
| `sudo` | 密码输入、关闭回显 | ✅ |

```java
public class PtyManager {
    private PtyProcess pty;
    private final Set<WebSocketSession> sessions = ConcurrentHashMap.newKeySet();

    // 环形缓冲区：保留最近 N 个输出块用于重连回放
    private final byte[][] ringBuffer = new byte[500][];
    private int ringHead = 0;  // 下一个写入位置
    private int ringCount = 0; // 已填充块数（≤ 500）

    // 启动 shell
    public void spawn(String shell, Path workDir, int cols, int rows) {
        pty = PtyProcess.exec(
            new String[]{shell},
            Map.of("TERM", "xterm-256color"),
            workDir.toFile()
        );
        pty.setWinSize(new WinSize(cols, rows));

        // 虚拟线程读取 stdout
        Thread.startVirtualThread(() -> {
            byte[] buf = new byte[8192];
            InputStream in = pty.getInputStream();
            int n;
            while ((n = in.read(buf)) != -1) {
                byte[] chunk = Arrays.copyOf(buf, n);

                // 写入环形缓冲区
                ringBuffer[ringHead] = chunk;
                ringHead = (ringHead + 1) % 500;
                if (ringCount < 500) ringCount++;

                // 广播给所有已连接客户端
                String b64 = Base64.getEncoder().encodeToString(chunk);
                broadcast(new TermOutputMessage(b64));
            }
        });
    }

    // 获取缓冲区全部内容（用于重连回放）
    public byte[] getReplayData() {
        if (ringCount == 0) return new byte[0];

        ByteArrayOutputStream out = new ByteArrayOutputStream();
        int start = (ringCount < 500) ? 0 : ringHead; // 满时从最老开始
        for (int i = 0; i < ringCount; i++) {
            byte[] chunk = ringBuffer[(start + i) % 500];
            if (chunk != null) out.write(chunk);
        }
        return out.toByteArray();
    }

    public void write(byte[] data) { pty.getOutputStream().write(data); }
    public void resize(int cols, int rows) { pty.setWinSize(new WinSize(cols, rows)); }
    public void destroy() { pty.destroy(); }
    public boolean isAlive() { return pty.isAlive(); }
}
```

关键点：
- **环形缓冲区**：500 块 × 8KB ≈ 4MB 上限，断线重连时回放历史输出
- **虚拟线程**读取 stdout，阻塞 I/O 不占用平台线程
- **广播模式**：一个 PTY 输出 → 所有已连接会话
- **关闭钩子**：`Runtime.getRuntime().addShutdownHook()` 确保 JVM 退出时杀子进程

public class FileHandler {

    private Path resolveSafe(String userPath) throws SecurityException {
        Path resolved = workspaceRoot.resolve(userPath).toRealPath();
        Path rootReal = workspaceRoot.toRealPath();
        if (!resolved.startsWith(rootReal)) {
            throw new SecurityException("Path traversal denied: " + userPath);
        }
        return resolved;
    }

    public List<FileEntry> list(String path) { ... }
    public String read(String path) { ... }

- `toRealPath()` 解析所有符号链接后再判断前缀

### AuthManager.java — 认证

```java
public class AuthManager {
    private final String token;

    public AuthManager(String token) {
        this.token = (token != null && !token.isEmpty())
            ? token
            : UUID.randomUUID().toString().replace("-", "").substring(0, 12);
    }

    public boolean validate(String input) {
        return MessageDigest.isEqual(
            token.getBytes(StandardCharsets.UTF_8),
            input.getBytes(StandardCharsets.UTF_8)
        );
    }
}
```

- 常量时间比较防止时序攻击
- 未配置 token 时自动生成 12 位随机码
- 支持三种配置来源：命令行参数 > 环境变量 > 自动生成

### terminal.js — 终端

- 初始化 xterm.js Terminal 实例，配置 `xterm-addon-fit` 自适应容器
- `term.onData(callback)` → Base64 编码 → WS 发送 `term.input`
- `window.onresize` → fit → WS 发送 `term.resize`
- 收到 `term.output` → Base64 解码 → `term.write(data)`
- 收到 `term.replay` → Base64 解码 → `term.reset()` 清屏 → `term.write(data)` 完整回放历史输出

### filemanager.js — 文件管理器

- 递归树形结构渲染，展开/折叠目录
- 右键菜单（桌面）或长按（移动端）：新建文件/文件夹、重命名、删除、下载
- 文件点击 → 请求 `fs.read` → 弹出编辑 Modal
- 拖拽文件到文件树 → FileReader 读取 → `fs.write` 上传
- 移动端底部抽屉式弹出

### app.js — 连接管理

- WebSocket 连接状态机：`disconnected → connecting → authenticating → connected`
- 首次连接 `term.init` 带 `fresh: true`
- 断线自动重连：检测 `onclose`/`onerror`，指数退避 + 随机抖动
- 重连时 `term.init` 带 `fresh: false`，触发服务端回放
- token 缓存在内存中，重连时自动发送 auth
- 最大重试无上限（封顶 15s 间隔），连接恢复前持续尝试

## 安全模型

```
                    ┌─────────────┐
                    │  WS 连接建立  │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  发送 auth   │
                    │  token验证   │
                    └──┬───────┬──┘
                       │       │
                  失败  │       │  成功
                 ┌─────▼─┐  ┌──▼────────┐
                 │ 断开   │  │ 已认证会话  │
                 │ 连接   │  │           │
                 └───────┘  │ 所有消息   │
                            │ 都经过     │
                            │ resolveSafe│
                            │ 路径校验   │
                            └───────────┘
```

**三层防护：**
1. **连接层** — token 认证，拒绝未授权访问
2. **路径层** — 所有文件操作强制解析符号链接后做前缀检查
3. **内容层** — 敏感文件读取警告，大文件截断保护

**不做的：**
- 不做 HTTPS（依赖反向代理或 Cloudflare Tunnel 提供 TLS）
- 不做用户体系（单 token，个人使用场景）
- 不做操作日志（保持轻量）

## 跨网络方案对比

| 方案 | 安装要求 | 注册 | HTTPS | 适用场景 |
|------|---------|------|-------|---------|
| Cloudflare Tunnel | 服务端装 cloudflared | 否 | 自动 | 通用推荐 |
| bore | 服务端装 bore | 否 | 否 | 极简场景 |
| localtunnel | 服务端有 Node | 否 | 自动 | 前端开发者 |
| Tailscale | 双端安装 | 是 | 自动 | 稳定组网 |
| frp | 需公网 IP VPS | 否 | 手动 | 自建基础设施 |

推荐 Cloudflare Tunnel 作为 README 首选方案，一行命令、零注册、免费 HTTPS、简历上 Cloudflare 生态也有含金量。
